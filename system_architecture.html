2.3 — Collaborative Filtering Engine (Item-Based kNN)

The CF engine powers two features: candidate verification AND skill autosuggest for users.

Mathematical Formulation

The CF algorithm computes a weighted average prediction for each unknown skill:

                    Σᵢ  similarity(candidate, skillᵢ) × rating(skillᵢ)
predicted_score = ─────────────────────────────────────────────────────
                    Σᵢ  |similarity(candidate, skillᵢ)|

Where:

  • candidate = a skill the user hasn't marked yet
  • skillᵢ iterates over the user's known skills
  • rating(skillᵢ) = +1 if user has it, -1 if user denied it
  • similarity(A, B) = precomputed cosine similarity from Stack Overflow data
Note

The cosine similarity for binary co-occurrence vectors reduces to:
sim(A, B) = |users_who_have_both| / √(|users_who_have_A| × |users_who_have_B|)
This is computed over 87,000+ real developer profiles, making it statistically robust.

3. Theoretical Explanation of the Entire System

3.1 — The Core Problem

Traditional skill-matching systems use keyword search: a recruiter types "React" and candidates with "React" appear. This fails when:

  1. Semantic Gap: Recruiter says "build a real-time dashboard" — no candidate has a skill called "real-time dashboard"
  2. Incomplete Profiles: A React developer almost certainly knows JavaScript, but may not have listed it
  3. Discovery: The system can't surface candidates who might have a skill but haven't been asked

3.2 — The AIM Solution Architecture

AIM (AI-Intelligent Matching) solves all three problems through a three-engine architecture:

Engine 1: NLP Skill Extraction (skillExtractor.js + api-server.js)

Purpose: Bridge the semantic gap between human language and technical skill names.

How it works:

  • The recruiter's natural language query is sent to Groq Cloud running the qwen3-32b large language model
  • The LLM receives a system prompt containing all 227 valid skill names from the dataset
  • It returns a structured JSON array of skills with confidence scores and reasoning
  • A 3-tier cascade optimizes for speed: exact match (0ms) → fuzzy match (~1ms) → NLP inference (~1-3s)
  • Validation layer ensures the LLM never hallucinates skill names — every returned skill must exist in the 227-skill dataset

Theoretical Basis: Transformer-based language models encode semantic relationships between concepts. The model understands that "build a real-time data pipeline" implies skills like Apache Kafka, Apache Spark, and Python — even though none of those words appear in the query verbatim.

Engine 2: Collaborative Filtering (cfEngine.js)

Purpose: Predict what other skills a user likely has based on their current profile.

How it works:

  • Offline phase: The precompute_similarity.py script processes the Stack Overflow 2023 Developer Survey (87,000+ developers). For each pair of skills, it computes cosine similarity based on co-occurrence:
    sim(A, B) = |developers_who_have_both| / √(|devs_with_A| × |devs_with_B|)
  • Runtime phase: For a user with skills {React, CSS}, the engine scans all 227 candidate skills and computes a weighted prediction score using the formula:
    prediction(X) = Σ(sim(X, known_i) × rating_i) / Σ|sim(X, known_i)|
    Where rating_i = +1 for confirmed skills and -1 for denied skills.

Theoretical Basis: This is Item-Based k-Nearest Neighbors Collaborative Filtering — the same algorithm used by Amazon for product recommendations ("customers who bought X also bought Y"). Here, "items" are skills and "users" are developers. The insight: if 70% of developers who know React also know JavaScript, then a React developer who hasn't listed JavaScript probably still knows it.

Key Innovation — Negative Signals: When a user clicks "No, I don't have this skill", it's recorded as a -1 rating. This actively suppresses related predictions. If Alice denies knowing TypeScript, the engine reduces predictions for Angular and other TypeScript-adjacent skills. This prevents the system from repeatedly suggesting irrelevant skills.

Engine 3: Relevance Engine (relevanceEngine.js)

Purpose: Decide which candidates should receive verification questions for a skill they haven't listed.

How it works: A dual-path scoring system combines two knowledge sources:

Path Source Coverage Precision
Path 1: Curated Graph skillGraph.js (~20 skills) Low Very High
Path 2: CF Fallback skill_similarity.json (227 skills) High High

The scoring hierarchy:

Relationship Score Example
Direct link in skill graph 10 React → JavaScript (explicitly linked)
CF similarity > 0.5 10 Python → NumPy (co-occur in 50%+ of profiles)
Shared neighbor (distance-2) 5 React → Angular (both link to JavaScript)
CF similarity > 0.3 5
Same domain 3 React → Vue.js (both "Frontend")
CF similarity > 0.15 3
No relationship 0 React → Cooking

If the maximum score across all of a user's skills ≥ 3, a verification question is generated.

Theoretical Basis: This is a hybrid knowledge graph + statistical inference system. The curated graph provides editorial precision (a human explicitly said "React is related to JavaScript"), while the CF data provides statistical coverage (the data shows React developers also tend to know Redux, even if nobody hand-coded that relationship).

3.3 — State Management & Persistence

The application uses React Context + useReducer — a lightweight, Redux-like pattern:

  • Single Source of Truth: All state lives in one AppContext object
  • Immutable Updates: Every action returns a new state object (no mutations)
  • Persistence: An useEffect hook serializes the entire state to localStorage on every change
  • Hydration: On app load, state is restored from localStorage (or initialized from defaults)
  • 7 Action Types: SET_RECRUITER_SEARCH, UNIFIED_SEARCH, ANSWER_QUESTION, ADD_SKILL, DENY_SKILL, SWITCH_USER, RESET_DATA

3.4 — The Skill Matcher Microservice

This is a separate, production-grade subsystem using a fundamentally different approach:

Feature aim_profile_app skill_matcher
Skill storage In-memory JSON PostgreSQL with pgvector
Search method Co-occurrence statistics Dense vector similarity (semantic)
Skill structure Flat list (227 skills) Hierarchical (Concept → SubSkill → MicroSkill)
AI model Groq cloud LLM Local sentence-transformers
Use case Recruiter ↔ candidate matching Skill taxonomy & semantic search

How it works:

  1. When a new skill is created via the API, the EmbeddingService encodes its name + description into a 384-dimensional vector using all-MiniLM-L6-v2
  2. This vector is stored alongside the skill in PostgreSQL using the pgvector extension
  3. Search queries are also encoded into vectors
  4. pgvector computes cosine distance between the query vector and all stored skill vectors
  5. Results are returned sorted by similarity, with their full hierarchical path

Theoretical Basis: Unlike the co-occurrence-based CF approach, this system uses dense semantic embeddings — continuous vector representations of text meaning. Two pieces of text with similar meaning (even if they share no words) will have vectors close together in 384-dimensional space. This enables queries like "my model keeps memorizing training data" to match the micro-skill "Overfitting" — because the transformer model understands the semantic relationship.

6. Key Design Decisions

Decision Rationale
3-tier extraction cascade Avoid ~$0.001/query LLM costs and 1-3s latency for simple queries like "React"
Pre-computed similarity 227×227 cosine matrix computed once offline; runtime is O(n) lookup, not O(n²)
Dual-path relevance High-precision manual graph for common skills + high-coverage statistical data for the long tail
Client-side CF No backend required for autosuggest — runs entirely in-browser from a 192KB JSON file
localStorage persistence Zero-infrastructure demo — no database required for the primary app's state
Groq + qwen3-32b Free-tier API with fast inference (~1s); qwen3-32b provides strong structured JSON output
Separate skill_matcher Production architecture pattern — demonstrates pgvector + embeddings for future scale
